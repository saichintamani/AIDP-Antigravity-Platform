# Phase 17: Adaptive Learning Layer

## 1. Mission Statement
Convert the static knowledge generated by the Scientific Introspection Engine (Phase 16) into dynamic policy updates that actively change the system's behavior. AIDP must transition from merely observing failure modes to physically adapting its parameters in a measurable, auditable, and reversible way.

## 2. Core Deliverables

### 2.1 Adaptation Record
An immutable, auditable log of every system adaptation. It records what component was changed, why it was changed (tracing back to an introspection lesson), and the exact before/after state of the parameter.

### 2.2 Adaptive Policy Registry
A central configuration store holding dynamic system weights. Instead of hardcoding defaults, agents and verifiers will query this registry at runtime to retrieve parameters such as:
- Reviewer persona weights
- Verification check priorities
- Prior confidence of biological assumptions

### 2.3 Adaptive Learning Engine
The active feedback loop. It ingests Observatories (e.g., Reviewer Precision, Assumption Tracking) and emits `AdaptationRecord` objects that update the Policy Registry.
- Example: If ReviewerPrecision drops below 0.5, emit a record lowering their weight.
- Example: If a FailureMode occurs frequently, emit a record increasing the priority of that check.

## 3. Architecture
- **AdaptationModels**: Contains the `AdaptationRecord` Pydantic model.
- **PolicyRegistry**: Holds current live weights and manages the application/rollback of records.
- **AdaptiveLearningEngine**: Computes deltas between the current policy state and the ideal state based on introspection data, generating new adaptation records to bridge the gap.
