# Platform Meta-Architecture

## Introduction
The Platform Meta-Architecture acts as the definitive set of "laws of physics" for the Artificial Intelligence Discovery Platform (AIDP). As we transition into detailed subsystem design, this document guarantees that the disparate components—ranging from mathematical solvers to Ray orchestration clusters—operate under unified constraints. No subsystem is permitted to deviate from the lifecycles, state models, or integration rules defined herein without a formal Engineering Decision Record (EDR).

## 1. Universal Execution Lifecycle
Every active process within AIDP (Agent Workflows, ML Training Jobs, Graph Ingestion Tasks) must strictly adhere to the following Universal Execution Lifecycle state machine:

1.  **`INIT`**: The process allocates necessary dependencies (e.g., pulling model weights into RAM, acquiring Ray worker locks) but mutates zero global state.
2.  **`VALIDATE`**: Pre-flight checks. The process verifies mathematically that the upcoming operation is well-formed (e.g., verifying matrix dimensions, checking JWT auth scopes).
3.  **`EXECUTE`**: The core operation. This phase must be highly observable and interruptible.
4.  **`CHECKPOINT`**: If the execution is long-running (e.g., multi-day ML training or multi-hour agent research), it must persist intermediate state to Ephemeral Storage (Redis) continuously.
5.  **`COMMIT / ROLLBACK`**: If successful, outputs are atomically written to Immutable Storage (Neptune/Qdrant/S3). If a failure occurs, the process rolls back cleanly, leaving no orphaned graph edges or zombie GPU processes.
6.  **`TERMINATE`**: Explicit resource deallocation and generation of the final OpenTelemetry trace span.

## 2. Platform State Model
AIDP distinguishes strictly between Ephemeral State and Immutable State to ensure reproducibility and distributed scalability.

*   **Immutable State (The Truth):**
    *   **Substrates:** S3 (Raw Documents), Amazon Neptune (Knowledge Graph), Qdrant (Vectors).
    *   **Rule:** Once a fact, document, or embedding is committed, it is *never* updated in place. Modifications are handled as append-only revisions with strict monotonically increasing timestamps. This enables time-travel querying and mathematical reproducibility of past experiments.
*   **Ephemeral State (The Scratchpad):**
    *   **Substrates:** Amazon ElastiCache (Redis), Ray Plasma Object Store.
    *   **Rule:** Used for agent conversation history, KV caches for LLM inference, and in-flight calculation tensors. If this data is lost due to a node crash, it must be safely re-computable or resume from a `CHECKPOINT`.
*   **MLOps State & Lineage:**
    *   **Substrates:** MLflow/WandB (Model Registry), Delta Lake (Dataset Versioning).
    *   **Rule:** Every embedding generated must carry metadata linking it back to the exact Git commit of the model pipeline, the specific dataset version used for fine-tuning, and the model hash. This prevents catastrophic drift if an underlying embedding model is swapped.

## 3. Interaction Contracts
To prevent integration chaos across distributed microservices:

*   **Strict IPC:** Inter-Process Communication (IPC) across distinct subsystems (e.g., the Agent Core talking to the Graph Engine) MUST use **gRPC**. 
*   **Schema Definition:** All payloads MUST be defined via **Protocol Buffers (Protobuf)**. The transmission of unstructured JSON objects between critical boundaries is strictly forbidden to ensure type-safety and backward compatibility.
*   **Asynchronous Messaging:** Decoupled, high-throughput events (e.g., "New Paper Ingested") MUST be published to an event bus (e.g., Amazon MSK / Kafka) using the CloudEvents specification.

## 4. Trust Boundaries & Security Enclaves
AIDP enforces a Zero-Trust architecture internally.

*   **The Execution Enclave:** Any code generated by an AI Agent (e.g., Python scripts for statistical analysis) is deemed "Hostile." It MUST execute within a hardware-isolated microVM (AWS Firecracker). 
    *   *Blast Radius Limit:* Firecracker microVMs operate with strict IAM least-privilege (no AWS API access), zero network egress (air-gapped from the VPC), and explicitly mounted read-only data volumes. A hypervisor escape must not yield access to the control plane.
*   **The Control Plane:** The orchestration services (Ray Head Node, API Gateways) run in highly trusted Kubernetes namespaces. They never execute arbitrary code.
*   **Identity Propagation:** Service-to-service authentication relies on **SPIFFE/SPIRE** for cryptographic identity issuance, passing short-lived JWTs within gRPC metadata. Securing the pipe via Istio mTLS is insufficient; every request payload must carry verifiable identity claims regarding *which* agent is invoking the action.

## 5. Cross-Cutting Concerns
*   **Idempotency:** Every API endpoint and task execution must be idempotent. A Ray worker failing and restarting a task three times must yield the exact same graph state as a task succeeding on the first try.
*   **Rate Limiting & Backpressure:** Every subsystem must implement circuit breakers. If the Neptune Graph DB is overwhelmed, it must return `HTTP 429` (or gRPC `RESOURCE_EXHAUSTED`), and upstream orchestrators must apply exponential backoff.
*   **Dead Lettering:** Any task that fails all retry attempts is routed to a Dead Letter Queue (DLQ) in S3 for forensic analysis. Data is never silently dropped.

## 6. Observability & SRE Strategy
AIDP mandates **OpenTelemetry (OTel)** as the universal standard and enforces strict Site Reliability Engineering (SRE) paradigms.

*   **Distributed Tracing:** A `Trace-ID` is generated at the API Gateway and passed through every gRPC context and Kafka header. If an Agent generates a hypothesis, that hypothesis carries the Trace-ID back to the original user prompt.
*   **Service Level Indicators (SLIs) & Objectives (SLOs):** Every subsystem MUST define its operational bounds. For example:
    *   *SLI:* 99th percentile response time for vector insertions.
    *   *SLO:* 99.9% success rate under 50ms latency.
*   **Error Budgets:** If a subsystem exhausts its 30-day error budget (e.g., due to downstream database timeouts), automated CI/CD pipelines halt feature deployments for that subsystem until stability is restored.
*   **Dashboards & Alerts:** Every subsystem must provision standard Grafana dashboards as code (Infrastructure as Code). Dashboards must track the Four Golden Signals: Latency, Traffic, Errors, and Saturation (including VRAM saturation).
*   **Logs & Metrics:** All logs must be structured JSON (including `Trace-ID`). `/metrics` endpoints must expose Prometheus formats, including mathematical confidence distributions and drift metrics.

## 7. AI Evaluation Checkpoints
Before any major state transition (e.g., an Agent finalizing a hypothesis and writing it to the Graph), the system must hit a mandatory Evaluation Checkpoint.

*   **Queueing & Backpressure:** The Evaluation Service operates under strict Little's Law constraints. To prevent infinite queuing delays for heavy metrics (e.g., computing KL-divergence via LLMs), token-bucket rate limiting will reject upstream requests (`RESOURCE_EXHAUSTED`) during traffic spikes, forcing agents to backoff and wait.
*   **Mandatory AI Metrics:** The checkpoint computes a standardized suite of evaluations before permitting the `COMMIT`:
    1.  **Hallucination Detection:** Cross-referencing generated assertions against immutable source vectors.
    2.  **Calibration Metrics:** Does the agent's stated confidence match the mathematical probability of the generated graph path?
    3.  **Citation Quality:** Are all referenced UUIDs valid, reachable, and logically coherent?
    4.  **Novelty / KL-Divergence:** Mapping back to `004_MATHEMATICAL_CAPABILITY_MAPPING.md` to ensure the hypothesis isn't mathematically trivial.
*   **Rejection:** If the assertion fails the threshold, it is kicked back to the `EXECUTE` phase for self-correction.

## 8. Subsystem Integration Rules (Dependency Order)
Subsystems must be constructed in a strict Dependency Order. A higher-level subsystem cannot be built until its dependencies are mocked or implemented via a defined Protobuf contract.

**Build Sequence:**
1.  **Execution & Orchestration Core** (Ray, Kubernetes foundations, OTel integration).
2.  **Knowledge Substrate** (Graph + Vector Databases, Ingestion endpoints).
3.  **Mathematical Engine** (Spectral clustering services, Bayesian updaters).
4.  **Agent Ecosystem** (The actual LLM workflows relying on all the above).
