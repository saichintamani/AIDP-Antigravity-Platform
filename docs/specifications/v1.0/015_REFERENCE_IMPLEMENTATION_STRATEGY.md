# Detailed Architecture: Reference Implementation Strategy

## Introduction
Before selecting exact ML frameworks, we must analyze the state-of-the-art industrial systems that have successfully scaled AI. This specification evaluates the top-tier libraries in the open-source ecosystem, documenting what problems they solve, their trade-offs, and whether AIDP will **Adopt**, **Adapt**, or **Reject** them.

---

## 1. High-Throughput Inference
### 1.1 vLLM vs. TensorRT-LLM
*   **Problem Solved:** LLM inference is memory-bound by the KV cache, leading to severe fragmentation and low batch sizes.
*   **Analysis:** **vLLM** introduced PagedAttention, treating KV cache like virtual memory (pages and blocks), nearly eliminating fragmentation. **TensorRT-LLM** offers lower-level Nvidia-specific graph fusions and kernel optimizations.
*   **Trade-offs:** TensorRT-LLM provides maximum speed on H100s but requires a brittle AOT (Ahead-Of-Time) compilation step. vLLM provides massive throughput with highly flexible Python APIs, crucial for MCTS rollouts where batch sizes fluctuate wildly.
*   **Decision:** **Adopt vLLM.** The dynamic batching flexibility is mandatory for the unpredictable workloads generated by the Cognitive Agent's planning algorithms.

---

## 2. Distributed Memory & Parallelism
### 2.1 DeepSpeed vs. PyTorch FSDP
*   **Problem Solved:** 70B+ parameter models cannot fit on a single GPU. Gradients, optimizer states, and parameters must be sharded across the cluster.
*   **Analysis:** **DeepSpeed (ZeRO-3)** shards all model states and handles NVLink communication efficiently. **PyTorch FSDP (Fully Sharded Data Parallel)** is deeply integrated into the native PyTorch ecosystem, offering similar sharding with cleaner backward-compatibility.
*   **Trade-offs:** DeepSpeed can be intrusive to the model code and suffers from version-lock issues. FSDP requires newer PyTorch versions but offers a vastly cleaner API. *Crucially, FSDP introduces massive All-Gather network latency, which will violate the MCTS 100ms planning SLA.*
*   **Decision:** **Adopt PyTorch FSDP, strictly limited to the primary 70B causal models.** The distilled MCTS Actor-Critic networks are explicitly exempted from FSDP and must run purely via single-node Tensor Parallelism (TP) to guarantee zero network latency during rollouts.

---

## 3. Hardware-Aware Attention Kernels
### 3.1 FlashAttention vs. xFormers
*   **Problem Solved:** Standard attention is $O(N^2)$ in time and memory, requiring massive HBM read/writes. 
*   **Analysis:** **FlashAttention** computes attention exactly while tiling blocks to keep data in SRAM, drastically reducing HBM bandwidth. **xFormers** provides a library of building blocks including FlashAttention variants.
*   **Trade-offs:** FlashAttention is a low-level kernel that can be difficult to compile across different CUDA versions.
*   **Decision:** **Adopt FlashAttention (via PyTorch SDPA).** PyTorch 2.x natively incorporates Scaled Dot Product Attention (SDPA) which uses FlashAttention under the hood, giving us the performance without the compilation nightmares.

---

## 4. Graph Computation
### 4.1 PyTorch Geometric (PyG) vs. DGL (Deep Graph Library)
*   **Problem Solved:** Variational Message Passing (VMP) requires massive sparse matrix multiplications over the Knowledge Substrate.
*   **Analysis:** **PyG** uses a scatter/gather methodology that is highly Pythonic. **DGL** treats graphs as first-class objects and optimizes sparse operations closer to the C++ level.
*   **Trade-offs:** PyG is easier to prototype but can struggle with graphs containing billions of edges. DGL scales better to multi-node training but has a steeper learning curve and a fragmented API.
*   **Decision:** **Adapt PyG.** AIDP operates on local Markov Blankets (via PPR), meaning the graph is naturally chunked into subgraphs. PyG is perfect for subgraph-level batching.

---

## 5. Distributed Orchestration
### 5.1 Ray vs. KubeRay
*   **Problem Solved:** MCTS and VMP must be distributed across hundreds of CPU and GPU nodes dynamically.
*   **Analysis:** **Ray Core** provides primitive Actor/Task abstractions. **KubeRay** maps Ray clusters onto Kubernetes natively.
*   **Trade-offs:** Raw Ray requires manual cluster management. KubeRay adds Kubernetes overhead but enables declarative autoscaling and precise hardware isolation (taints/tolerations).
*   **Decision:** **Adopt KubeRay with Strict Node Pool Isolation.** To prevent vLLM's aggressive PagedAttention from starving PyG's message-passing matrices and causing OOM crashes, KubeRay is mandated to isolate vLLM strictly to `$p5` inference pools, while PyG executes on distinct `$r7a` CPU-heavy pools, communicating over the Ray object store.

---

## 6. Decision Register

### Accepted Decisions
*   **KubeRay with Strict Node Isolation:** Adopted. vLLM and PyG must be placed on entirely separate physical nodes via Kubernetes taints/tolerations to prevent CUDA OOM starvation.
*   **Exempting MCTS from FSDP:** Adopted. MCTS Actor-Critic networks must use single-node TP to bypass the All-Gather network latency of FSDP.
*   **vLLM** for LLM inference (mandatory for dynamic MCTS batching).
*   **PyTorch FSDP** for model sharding of massive 70B+ models.
*   **PyTorch SDPA (FlashAttention)** for $O(N^2)$ bottleneck mitigation.
*   **PyTorch Geometric (PyG)** for VMP graph execution.

### Rejected Alternatives
*   **TensorRT-LLM:** Rejected due to brittle AOT compilation pipelines.
*   **DeepSpeed:** Rejected due to API intrusiveness for custom dual-process models.
*   **DGL:** Rejected. PyG's subgraph batching aligns perfectly with our PPR context retrieval.

### Open Questions
*   Does vLLM's PagedAttention support customized prefix-caching across distinct Ray Actors, or is the cache strictly bound to a single vLLM instance?

### Risks
*   **vLLM Memory Overhead:** PagedAttention pre-allocates a massive chunk of HBM. This could starve the PyG message-passing algorithms if they share the same GPU.

### Validation Plan & Success Metrics
*   **Validation:** Profile GPU memory allocation on a single $p4d.24xlarge$ node running both vLLM and PyG simultaneously.
*   **Metric:** Avoid Out-Of-Memory (OOM) errors during a simulated 100-node MCTS rollout.
