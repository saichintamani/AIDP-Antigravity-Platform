# High-Level System Architecture (HLSA)

## 1. Architecture Vision & Traceability Context
The Artificial Intelligence Discovery Platform (AIDP) is engineered as a highly decoupled, distributed, and scalable system centered entirely around deep reasoning and rigorous scientific validation. Aligning with the Project Constitution, this architecture prioritizes epistemic integrity over real-time processing and intentionally targets AWS as its sole cloud provider (avoiding multi-cloud abstractions). 

Crucially, every architectural decision below is explicitly mapped to the System Requirements Specification (SRS) via the **Requirements Traceability Matrix (RTM)**, ensuring that no module exists without a justified purpose.

---

## 2. Global Data Flow (The Discovery Pipeline)
The macro-architecture follows a directed, cyclical pipeline mapping data ingestion to knowledge synthesis and hypothesis testing.
*   **Stage 1: Ingestion & Parsing** (Satisfies `REQ-KA-001`, `REQ-KA-004`)
    *   Continuous scrapers pull literature from arXiv, PubMed, and custom endpoints into raw S3 buckets.
    *   Text extraction and NLP pipelines break down unstructured PDFs into textual chunks and tabular data.
*   **Stage 2: Knowledge Extraction** (Satisfies `REQ-KR-001`, `REQ-KR-004`)
    *   LLM-based entity and relation extraction models process the chunks, outputting Subject-Predicate-Object triplets with confidence scores.
*   **Stage 3: Graph Integration** (Satisfies `REQ-KG-001`, `REQ-KG-004`)
    *   Triplets are ingested into the Core Knowledge Graph (Amazon Neptune / Neo4j) while vector embeddings of the text are stored in a linked Vector Database (Milvus/Qdrant).
*   **Stage 4: Autonomous Reasoning** (Satisfies `REQ-AR-001`, `REQ-SD-001`)
    *   Research Agents query the graph, running multi-hop traversals to deduce missing links or formulate novel hypotheses.
*   **Stage 5: Experimental Validation** (Satisfies `REQ-ML-002`, `REQ-ET-001`)
    *   Hypotheses trigger automated ML training runs or mathematical simulations. The output metrics and validated conclusions are written *back* into the knowledge graph, completing the cycle.

## 3. Cognitive Architecture & Memory Systems
The intelligence of AIDP is not a single monolith, but a composable cognitive architecture enabling complex reasoning.

### 3.1 Core Reasoning Engine (`REQ-AR-003`, `REQ-AR-005`)
Instead of a simple ReAct loop, the Core Reasoning Engine implements an advanced *Tree of Thoughts (ToT)* or *Graph of Thoughts (GoT)* orchestrator. 
*   **Hypothesis Generator:** Proposes N possible answers/hypotheses to a query.
*   **Evaluator:** A secondary model critiques each node, pruning invalid paths by checking against the Knowledge Graph (`REQ-SD-002`).
*   **Traceability Logger:** Every branch of the reasoning tree is saved to memory, providing a complete explainability trace for the final output (`REQ-AR-005`).

### 3.2 Episodic & Semantic Memory (`REQ-RA-004`)
*   **Semantic Memory:** The global Knowledge Graph (long-term, immutable facts).
*   **Episodic Memory:** A Redis/DynamoDB layer storing the exact state, dialogue, and context window of a specific agent's research session, allowing long-running agents to resume work seamlessly.

---

## 4. Agent Ecosystem
AIDP operates as a Multi-Agent System (MAS) where specialized agents communicate via a secure internal message broker (e.g., Apache Kafka / RabbitMQ).

*   **Librarian Agent:** Specialized in querying the KG and Vector Search to build literature reviews (`REQ-KD-008`).
*   **Mathematician Agent:** Specialized in symbolic math and calling external math calculators (e.g., SymPy) to verify formulas (`REQ-MATH-002`, `REQ-AR-010`).
*   **Peer Reviewer Agent:** Operates adversarially to critique the hypotheses generated by other agents, detecting logical fallacies (`REQ-AR-007`, `REQ-SD-008`).

## 5. Knowledge Substrate (Graph & Vector Search)
The foundational data layer of AIDP physically separates relational logic from semantic similarity while keeping them logically intertwined.

### 5.1 The Dual-Database Approach
*   **Property Graph (Amazon Neptune or Self-hosted GraphDB):** 
    *   Stores entities, discrete relationships, temporal data, and confidence scores (`REQ-KR-003`, `REQ-KG-002`).
    *   Provides multi-hop traversal (`REQ-KD-001`).
*   **Vector Database (Milvus / Qdrant):** 
    *   Stores dense embeddings of text chunks and abstract concepts.
    *   Handles highly optimized Approximate Nearest Neighbor (ANN) and Hybrid search (`REQ-VS-001`, `REQ-VS-002`).

### 5.2 Fusion Layer
An API gateway that translates a user’s complex query into a fused operation:
1.  Perform ANN in Vector DB to find relevant node IDs.
2.  Use those node IDs as entry points for a Cypher/Gremlin traversal in the Graph DB.

## 6. Distributed Execution & ML Orchestration
AIDP treats AI model training and hypothesis testing as first-class citizens.

### 6.1 Training & Inference Clusters (`REQ-ML-002`, `REQ-CM-002`)
*   **Kubernetes (Amazon EKS):** The orchestration layer for all workloads. 
*   **Ray / PyTorch Distributed:** For distributed model training, workloads are submitted as Ray Jobs to a dynamically scaling autoscaling cluster utilizing Spot instances when possible (`REQ-CM-004`).
*   **vLLM / TensorRT Inference:** Model weights are pulled from S3 and hosted on dedicated inference pods with dynamic batching.

### 6.2 Experiment Tracking (`REQ-ET-001`, `REQ-ET-004`)
*   An internal MLflow or Weights & Biases (W&B) instance acts as the Model Registry (`REQ-ML-004`).
*   All metrics, loss curves, and artifact locations (S3) are strictly tracked per run.

---

## 7. AWS Deployment & Physical Infrastructure
In adherence with the Project Constitution, the system heavily leverages AWS cloud-native solutions to maximize reliability and minimize maintenance of boilerplate infrastructure.

### 7.1 Networking & Compute
*   **VPC:** Multi-AZ deployment spanning public and private subnets (`REQ-CM-003`).
*   **EKS:** Manages the entire application plane (Agents, APIs, ingestion workers).
*   **S3:** The Data Lake for raw PDFs, model weights, and system backups.

### 7.2 Storage & Databases
*   **Amazon Aurora (PostgreSQL):** For relational user data, RBAC, and platform state.
*   **Amazon ElastiCache (Redis):** For high-speed episodic memory and API rate limiting.
*   **Amazon Neptune:** Fully managed graph database.

---

## 8. Security Boundaries & Sandboxing
Security is not bolted on; it is architectural, anticipating the risks of autonomous agents executing code.

*   **API Security:** Amazon API Gateway handles all external traffic, verifying JWTs issued by Amazon Cognito or an internal Auth service (`REQ-SEC-001`, `REQ-SEC-002`).
*   **Agent Sandboxing (`REQ-SEC-005`):** 
    *   Agents often generate Python code to perform data analysis (`REQ-RA-006`).
    *   This code is NEVER executed on the host container. It is dispatched to isolated **gVisor** or **AWS Firecracker microVMs** with zero outbound network access and strict compute/memory quotas to prevent malicious exploits or resource exhaustion.
*   **Encryption:** KMS manages encryption keys. All data is encrypted at rest (S3, EBS) and in transit (mTLS within the EKS cluster) (`REQ-SEC-003`, `REQ-SEC-004`).

---

## 9. Observability & Telemetry
Deep traceability requires deep observability. 

*   **Distributed Tracing:** OpenTelemetry sidecars run on all pods, propagating Trace IDs across API calls, Agent invocations, and Database queries. This allows debugging exactly where an agent's reasoning bottlenecked (`REQ-OBS-xxx`).
*   **Metrics & Logging:** FluentBit aggregates logs to Amazon OpenSearch or Datadog. Prometheus handles time-series metrics for GPU utilization and API latency. 
*   **Cost Observability:** Every Kubernetes pod and AWS resource is strictly tagged (e.g., `TenantID`, `ExperimentID`), allowing granular cost attribution for long-running compute jobs.
